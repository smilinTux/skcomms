# SKFed First-Contact Consent — Design (research-grounded)

**Status:** Design. Coord epic `dc6b1996`. Source: deep-research `w97gx0be7`
(107 agents, 24 sources, 24/25 claims verified). Gate this BEFORE publicizing the
directory (`2de60cb4`).

## Principle (field-validated)

**Discoverability ≠ delivery.** Being listed/reachable in the public directory
must NOT equal being delivered or notified. This is the settled answer where
Signal and SimpleX — the two strongest primary-source models — converge.

We drew the PQ ratchet from **SimpleX**, which is spam-resistant *by construction*
(no user identifiers → no global address to target; contact only via out-of-band
one-time link). By **adding a public directory** we deliberately took the
*discoverable* branch (Signal's branch) and inherited its spammability — so we
**must reconstruct consent at the recipient node.** (Even SimpleX isn't absolute:
the "impossible by construction" framing was *refuted* 0-3 — public-group members
can DM.)

## The recommended model

### 1. Recipient-side quarantine — Signal "Message Request" (verified, primary)
Unknown first sender → **quarantined request queue**: stored, but
- no push notification, no ring on calls,
- no read/delivery receipts sent back (no presence/typing leak),
- blurred profile metadata, **no link rendering** on the request screen,
- capped (~1 message) until the recipient **accepts / deletes / blocks**.

Known/accepted → delivered with full features. Safe **OOTB default =
quarantine-by-default, accept-on-explicit-action**, with an optional auto-accept
opt-in (SimpleX has this too).

### 2. Per-contact unguessable capability token — Signal "sealed sender / delivery token" (verified)
**The mechanism I under-weighted.** Signal gates unsolicited messages on
proof-of-knowledge of a **96-bit delivery token derived from the recipient's
profile key, shared only with contacts** ("allow from anyone" is OFF by default;
the token can be regenerated to block someone).

Transfer: the directory advertises a **request endpoint** (anyone may send a
*contact request*), but **actual delivery requires a per-contact capability token
issued only on acceptance.** So a leaked/enumerated directory entry alone is
*insufficient to deliver* — it only lets you *knock*. Established contacts ride a
token-authenticated path; first-contact rides the rate-limited/PoW-gated knock path.

> Open question (flagged by the research): how to **issue/rotate** per-contact
> tokens in a *federated* directory where the entry itself is public, so
> revocation/blocking works without breaking legitimate contacts. This is the core
> design challenge to solve.

### 3. Identity tiering — anonymous vs sovereign (our RFC-0001 dual identity)
E2EE forces spam defense onto **behavioral/structural** signals (content is
opaque). Tier the gate:
- **Anonymous** sender → strictest: proof-of-work challenge + per-sender rate
  limit + capability-token required.
- **Sovereign (DID-signed) from a known realm**, or **introduced/vouched**
  (web-of-trust) → reduced friction.

> Open/unsettled: the literature gives **no canonical tiering policy** — concrete
> thresholds (PoW difficulty, rate limits, vouching depth) that price out bulk
> spam without hurting legitimate first-contact are an open question.

### 4. Group-join consent — SimpleX toolkit (verified, directly transferable)
Groups are **invite-only by default, never join-from-directory**:
- **Captcha admission** — owner can require an unknown joiner to pass a captcha
  generated by the directory bot itself (no 3rd-party servers). [SimpleX v6.3]
- **"Knocking" / member review** — admins/moderators vet every prospective member
  before they join. [SimpleX v6.4.1]
- **Delegated moderator role** — approve members in review, moderate, block-for-all.
- **Shadow-block** — hide a suspected attacker's messages from everyone while
  their own view is unchanged.

### 5. Consent-gated abuse reporting (MlsGov, medium confidence)
Only messages a user **explicitly reports** become visible to moderators;
unreported messages stay cryptographically private. Reports disclose only minimal
metadata (Signal: originating id + a one-time anonymous message id — never content
or social graph).

## Honest gaps (treat as open)
The verified evidence is **strong on Signal + SimpleX**, but the research surfaced
**no verified claims** for **Matrix** (Mjolnir/Draupnir, policy/ban lists, server
ACLs, invite-spam), **email** (SPF/DKIM/DMARC, greylisting, hashcash economics), or
**Nostr** (NIP-05, NIP-13 PoW, paid relays, web-of-trust). Those branches of the
question are unsupported here and need a follow-up pass before we lean on them.

## Build order (proposed)
1. Recipient-side **request queue** + quarantine semantics (no-notify, capped,
   no receipts) — the highest-value, lowest-risk piece; pure node-side.
2. **Capability-token** delivery path (issue-on-accept, token-authenticated send)
   + the directory advertising a *request* endpoint vs a *delivery* endpoint.
3. **Identity tiering** (anon PoW/rate-limit vs sovereign/introduced) + per-sender
   rate limit at the `:443` inbox.
4. **Group-join consent** (captcha + knocking + moderator + shadow-block).
5. Follow-up research pass on Matrix/email/Nostr; settle tiering thresholds.
